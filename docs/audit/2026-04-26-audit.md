# Agnostik Security Audit — 2026-04-26

> Pre-1.0.0 closeout pass. Inputs: full source review of `src/*.cyr`,
> external CVE/0-day search against pattern-equivalent libraries
> (cJSON, jq, Ruby JSON, npm semver, gofiber UUID).
>
> Auditor: Claude (Opus 4.7) under direction of Robert MacCracken.
> Scope: agnostik public API surface. Out of scope: Cyrius toolchain
> itself, vendored stdlib in `lib/`, downstream consumers.

## Threat model

Agnostik is a **type-vocabulary library** consumed by every AGNOS
component (daimon, hoosh, agnoshi, aegis, argonaut, sigil, ark, kavach,
stiva, nein, yukti). It defines, parses, and serializes shared types —
no domain logic. The attack surface is therefore narrow:

- **Deserializer side**: every `*_from_json(src)` and `*_from_str(s)`
  function takes external bytes and decodes them. A malformed input
  must not corrupt memory, leak data, or produce wildly wrong values
  that downstream consumers will trust.
- **Identifier generation**: UUID v4 generators (`agent_id_new`,
  `execution_id_new`, derived span/trace IDs) must produce
  unpredictable values — these are session and authorisation tokens
  to consumers like aegis and sigil.
- **Secret handling**: `secret_new` / `secret_destroy` must not leak
  the secret value via uninitialised memory, logging, or comparison
  side channels.

What is **out of agnostik's threat model**:

- No process spawning (no `execve` / `fork` / `clone` / `sys_system`
  in any source file — verified by `scripts/security-scan` in CI).
- No file write surface — agnostik only reads `/dev/urandom`.
- No network surface.
- No shell interpolation.

## Findings

### F-001 — UUID PRNG silently degrades to deterministic seed (HIGH)

**Location**: `src/types.cyr:1-26` (`_prng_init`, `_prng_next`, `agent_id_new`)

**Description**: The PRNG seeds a 64-bit xorshift state from an 8-byte
read of `/dev/urandom`. On open or read failure it falls through to
`clock_gettime(CLOCK_MONOTONIC)` (timing-derived, low entropy in
sandboxed envs). On further failure it falls through to the hardcoded
constant `0x853c49e6748fea9b` — fully deterministic across processes.

```
var fd = syscall(2, "/dev/urandom", 0, 0);
if (fd >= 0) {
    syscall(0, fd, &buf, 8);              # no return-value check
    syscall(3, fd);
    _prng_state = load64(&buf);
}
if (_prng_state == 0) {
    syscall(228, 4, &buf);                # clock_gettime fallback
    _prng_state = load64(&buf) ^ load64(&buf + 8);
}
if (_prng_state == 0) {
    _prng_state = 0x853c49e6748fea9b;     # deterministic fallback
}
```

**Impact**: agnostik UUIDs back `agent_id`, `execution_id`, `user_id`,
`session_id`, `trace_id`, and `span_id` across every AGNOS component.
In a misconfigured sandbox / container / FIPS-mode environment where
`/dev/urandom` is unavailable, two AGNOS processes can produce identical
ID sequences. Downstream consumers (aegis, sigil) treat these IDs as
unpredictable session and authorisation tokens. **This is the same
pattern as CVE-2025-66630.**

Additionally, even on the happy path, only 64 bits of entropy are
mixed into the state — every subsequent `agent_id_new()` produces a
128-bit UUID with effective 64-bit entropy (sufficient against
collision, weak against an attacker who can observe other IDs).

**Severity**: **HIGH** (silent degradation to predictable values in
realistic deployment scenarios; impacts authorisation primitives).

**Remediation** (planned for 1.0.0):

1. Read 16 bytes per `agent_id_new` call directly from `/dev/urandom`,
   not 8 bytes once at process start.
2. Check the return value of `read` — partial read or short return
   must propagate `Err`.
3. Remove the deterministic constant fallback. If urandom is
   unavailable, `agent_id_new` should return `Err` and the consumer
   must handle it explicitly.
4. Keep `getrandom(2)` (`syscall 318`) as the primary path with
   urandom-open as fallback; the syscall doesn't require an fd and
   is preferred in sandboxed envs.

### F-002 — JSON string parser ignores escape sequences (MEDIUM)

**Location**: `src/types.cyr:_json_str`

**Description**: `_json_str` finds string values by scanning forward
to the next `"` byte without tracking preceding `\`:

```
while (vi < slen && load8(data + vi) != 34) { vi = vi + 1; }
```

A JSON value containing an escaped quote (`"abc\"xyz"`) terminates
parsing at the escaped quote, returning `abc\`. The remaining bytes
shift the field-extraction window for any subsequent `_json_*` call
on the same input, producing wrong values for unrelated fields.

**Impact**: All `*_from_json` deserializers operate on text the
consumer has already accepted from external sources (LLM tool output,
config files, telemetry payloads). A maliciously-crafted string field
can corrupt the parse of *other* fields in the same document, causing
silently wrong agent IDs, capability sets, classification results,
or telemetry attributes. **Pattern match to CVE-2025-27788 (Ruby JSON
unescape OOB).** No memory-safety violation in agnostik (we don't write
into a fixed buffer), but cross-field corruption is a correctness flaw
the audit should not ship past.

**Severity**: **MEDIUM** (correctness; downstream consumer impact
depends on whether they re-validate parsed fields).

**Remediation**: Track `prev == '\\'` while scanning; treat `\"` as
literal. While in there, also unescape `\\`, `\n`, `\t`, `\r` and
the standard `\uXXXX` form. Roundtrip test the full set against
`_to_json` output.

### F-003 — `_json_int` accepts garbage as zero, no overflow guard (MEDIUM)

**Location**: `src/types.cyr:_json_int`

**Description**: The integer extractor:

```
while (vi < slen && parsing == 1) {
    var ch = load8(data + vi);
    if (ch >= 48 && ch <= 57) { val = val * 10 + (ch - 48); vi = vi + 1; }
    else { parsing = 0; }
}
return val;
```

Three issues: (a) no leading sign — JSON `-1` parses as `0`; (b) no
digit-count cap — a 25-digit input wraps the i64 silently and returns
garbage; (c) on `_json_find_value` miss the function returns `0`
indistinguishably from the literal value `0`.

**Severity**: **MEDIUM** — directly matches GHSA-72HV-8253-57QQ
(Jackson unchecked numeric lengths) and the cJSON CVE-2025-57052
pattern of "non-digit chars silently accepted as digits".

**Remediation**: (a) handle `-` prefix; (b) cap at 19 digits then
return `Err` on the 20th; (c) change return type to `Result(int)`
so missing key vs literal zero are distinguishable. Item (c) is a
**breaking API change** — appropriate for the 1.0.0 cut but must be
called out in the CHANGELOG migration guide.

### F-004 — `_json_find_value` matches key inside string values (LOW)

**Location**: `src/types.cyr:_json_find_value`

**Description**: The key-finding probe builds `"max_memory":` as a
needle and `strstr`s for it in the entire input. If a JSON string
*value* contains the literal text `"max_memory":`, the substring
match wins and points at the wrong place.

**Severity**: **LOW** — exploitation requires the attacker to control
a string field that the consumer feeds into a `_from_json` deserializer
expecting a known integer key.

**Remediation** — landed in 1.0.0. `_json_find_value` now walks the
input tracking whether the cursor is inside a `"..."` string,
respecting `\` escapes so an embedded `\"` doesn't end the string
prematurely. The needle is matched only at the top level. Two
regression cases in `tests/tcyr/test_audit_2026_04_26.tcyr`
(`test_f004_key_inside_string_value`, `test_f004_string_with_escaped_quotes`).

### F-005 — `version_from_str` accepts non-numeric segments (LOW)

**Location**: `src/types.cyr:version_from_str`

**Description**: `str_to_int("abc")` returns `0` (verified — agnostik's
own `str_to_int` does not signal failure). `version_from_str("abc.1.2")`
returns `Version{0,1,2}` rather than an error.

**Severity**: **LOW** — version strings are usually self-controlled
(read from manifest, hardcoded in source). Rarely accept-from-untrusted-
input. Not exploitable for memory safety; classified as garbage-in-
garbage-out correctness.

**Remediation**: Validate that each segment is wholly digits before
calling `str_to_int`. Already-existing `string_shift_bug.tcyr` test
should grow a `assert_err(version_from_str("abc.1.2"))` case.

### F-006 — `cyrius lint` flags hand-written UFCS-style serde fns (UPSTREAM)

**Location**: `cyrius lint` over `src/agent.cyr` `src/config.cyr`
`src/hardware.cyr` `src/llm.cyr` `src/telemetry.cyr` `src/validation.cyr`

Agnostik names its serde adapters `<PascalStructName>_<snake_verb>`
(e.g. `ResourceLimits_to_json`, `AgentInfo_from_json`). This is a
deliberate UFCS-style "method-on-type" convention — it pairs each
struct to its adapters by struct name and is the same shape that
`#derive(Serialize)` emits, so hand-written and derived adapters
mix without consumer-visible naming drift.

`cyrius lint`'s `fn name should be snake_case` rule fires against
every site (28 warnings across 6 modules in 1.0.0). The flagged
functions are hand-written, not derive codegen — verified against
the line numbers cyrlint reports, e.g. `src/agent.cyr:47` is a full
hand-written `fn ResourceLimits_to_json(ptr, sb) { ... }` body. The
rule is correctly pointing at real source; the rule itself is wrong
to flag this naming pattern.

**Severity**: **UPSTREAM** — not a defect in agnostik. Filed against
the Cyrius toolchain at
[`docs/development/issues/cyrius-lint-ufcs-pascal-prefix-snake-case-2026-04-26.md`](../development/issues/cyrius-lint-ufcs-pascal-prefix-snake-case-2026-04-26.md)
with a reproduction and a suggested fix: recognise
`^[A-Z][A-Za-z0-9]*_[a-z][a-z0-9_]*$` (Pascal prefix + snake suffix)
as a UFCS-style method pattern and skip the snake_case rule when the
leading PascalIdent resolves to an in-scope `struct` or `enum`.
A narrower "exempt derive codegen" fix (the first draft of this
report) would not clear any of agnostik's 28 warnings — the user
caught this misattribution and the report has been corrected.

Until the upstream fix lands, agnostik's CI tolerates these warnings
explicitly — they're catalogued as known false-positives in the
upstream issue, not suppressed in bulk, so a real new snake_case
violation would still surface for review.

### F-008 — `_json_int` allows i64 overflow on 19-digit inputs (LOW)

**Location**: `src/types.cyr:_json_int`

**Description**: F-003 capped the digit run at 19 to stop runaway
parsing. But i64::MAX = `9223372036854775807` is *also* 19 digits.
Any 19-digit input in `[9223372036854775808, 9999999999999999999]`
was accepted and silently wrapped via i64 overflow:

- Input: `{"x": 9999999999999999999}`
- Pre-fix return: `-8446744073709551617`
- Risk: a downstream guard like `if (total_tokens > LIMIT) deny()`
  on a non-negative-only counter would see the negative number and
  *pass*, letting the attacker bypass quota enforcement.

Pattern shape: same family as **GHSA-72HV-8253-57QQ** (Jackson
async unchecked numeric lengths) and CWE-190 (integer overflow).

**Fix**: pre-multiply overflow check using i64::MAX = 922337203685477580 × 10 + 7:

```cyrius
if (ch < 48 || ch > 57)                                   { parsing = 0; }
elif (val > 922337203685477580)                           { parsing = 0; }
elif (val == 922337203685477580 && (ch - 48) > 7)         { parsing = 0; }
else { val = val * 10 + (ch - 48); vi = vi + 1; }
```

The unused `digits` counter from F-003 is removed — the overflow
check supersedes it. Magnitude check is symmetric for negatives, so
i64::MIN is unrepresentable by 1; accepted as a deliberate trade
(no consumer relies on i64::MIN as a sentinel).

### F-009 — `_json_str` reads past end on truncated `null` literal (LOW)

**Location**: `src/types.cyr:_json_str`

**Description**: After locating the value, `_json_str` probed for
a `null` literal via `memeq(data + vi, "null", 4)`. `memeq` reads 4
bytes unconditionally — no bounds check on `vi + 4 <= slen`. On
truncated input like `{"x":nu}` (or any case where the value sits
within 3 bytes of the buffer end), the probe over-reads up to 3
bytes past `slen`.

Risk: bounded over-read. Bump-allocator info-leak shape (adjacent
heap might decide whether the function returns 0 or proceeds);
freelist-allocator near-page-boundary SIGSEGV shape. Severity LOW
because the read is bounded to 3 bytes and not directly observable
by the attacker — but it is a real bounds violation. Pattern shape:
**CVE-2025-27788** family.

**Fix**: gate the probe on remaining length:

```cyrius
if (vi + 4 <= slen && memeq(data + vi, "null", 4) == 1) { return 0; }
```

### F-010 — RFC 7159 whitespace handling (INFO)

**Location**: `src/types.cyr:_json_find_value`

**Description**: After matching the `:`, the value-skip loop only
advanced past space (`32`). RFC 7159 §2 defines JSON whitespace as
one of: space (32), tab (9), LF (10), CR (13). Inputs from any
RFC-compliant pretty-printer — `python3 json.dumps(indent=2)`,
`jq .`, or any consumer that emits a newline after a colon —
silently failed to find the value, returning 0. This is
indistinguishable from "the value really was 0", so a request that
*should* have set `max_memory: 268435456` and was pretty-printed
silently became `max_memory: 0` (unbounded).

Risk: parse correctness, not memory safety. Defense-in-depth + RFC
compliance. `_json_int` and `_json_str` benefit transparently
because they consume the post-skip `vi` from `_json_find_value`.

**Fix**: replace single-byte skip with the full RFC 7159 set:

```cyrius
var skipping = 1;
while (vi < slen && skipping == 1) {
    var w = load8(data + vi);
    if (w == 32 || w == 9 || w == 10 || w == 13) { vi = vi + 1; }
    else { skipping = 0; }
}
```

### F-011 — Silent dead-code: derive + hand-written serde collision (LOW)

**Location**: 9 sites across 6 files — `src/agent.cyr` (×4),
`src/config.cyr`, `src/hardware.cyr`, `src/llm.cyr`,
`src/telemetry.cyr`, `src/validation.cyr`.

**Description**: Each affected struct carried both
`#derive(Serialize)` and a hand-written `<Type>_to_json` /
`<Type>_from_json` adapter. Cyrius last-define-wins meant the
hand-written adapter always won; the derive codegen output was
silently dead — but the dead bytes still shipped in the binary.
Surfaced by Cyrius v5.7.9's new `duplicate fn '<name>' (last
definition wins)` warning as 18 hits.

Risk: dead-code shape, not CVE shape. But: real risk that a future
contributor *removes* the hand-written adapter assuming the derive
will pick up — and in that ordering the derive's output *does*
run, with subtly different escape/number behavior than the
hand-written one (which incorporated F-002 / F-003 / F-008 fixes).
Best to make the intent explicit at the source.

**Fix**: dropped the 9 `#derive(Serialize)` markers. The
hand-written adapters were already what the test suite exercised;
binary shrinks 254,680 → 211,768 B (-42,912 B / -16.8 %) confirming
the derive output was real shipped dead code.

### Pre-existing build warning fixed in this pass

### Identifier-generation cost (expected and accepted)

`agent_id_new`, `span_id_new`, and dependents now hit `getrandom(2)`
once per call instead of advancing a 64-bit xorshift state in CPU.
The cost trade-off is deliberate: per-call kernel entropy versus a
64-bit seed reused for every UUID in the process lifetime.

| Benchmark              | Pre-audit (xorshift) | Post-fix (CSPRNG) | Δ      |
|------------------------|----------------------|-------------------|--------|
| `agent_id_new`         | 65 ns                | 608 ns            | +9.4×  |
| `trace_context_new`    | 130 ns               | 1 µs              | +7.7×  |
| `trace_context_child`  | 50 ns                | 566 ns            | +11.3× |
| `agent_id_to_str`      | 210 ns               | 728 ns            | +3.5×  |
| `agent_id_roundtrip`   | 529 ns               | 1 µs              | +1.9×  |
| `traceparent_format`   | 755 ns               | 1 µs              | +1.3×  |
| `security_context_full`| 224 ns               | 720 ns            | +3.2×  |

Real-world impact: an AGNOS daimon spawning ~10 agents/second pays
6 µs/sec extra CPU. We accept that cost over predictable IDs.
