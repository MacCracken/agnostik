# Agnostik Security Audit — 2026-06-01 (v1.3.0 closeout pass)

**Pass type:** minor-cut audit per CLAUDE.md §"Security Hardening
(before every release)" + the v1.3.0 refactoring/optimization closeout
review.
**Scope:** all `src/*.cyr` at the v1.3.0 cut (post-1.2.3).
**Toolchain:** cyrius `6.0.26` (pin advanced `6.0.14` → `6.0.26` this
release).

This audit accompanies a refactoring/optimization review rather than a
new-feature surface. No new untrusted-input parsers landed since the
2026-05-10 (1.0.8) audit; the 1.1.x/1.2.x line added sub-byte widths,
serde derive, OTLP proto helpers, and toolchain refreshes. The review
re-walked every parser that touches external bytes and surfaced one
buffer-safety finding (F-013) plus a set of low-priority deferred
cleanups (dead-code clusters, setter-less flag getters, an
over-allocated struct) recorded under "Deferred".

## Threat model

Unchanged from prior audits. Agnostik is the shared types library
consumed by every AGNOS component (daimon, hoosh, agnoshi, aegis,
argonaut, sigil, ark, kavach, stiva, nein, yukti). The attack surface
is **untrusted JSON / version / UUID / traceparent input** crossing the
serde boundary into agnostik parsers; output flows back as
`Result<value, AgnostikError>` or typed structs. No syscall surface
interacts with external state beyond CSPRNG (`getrandom` /
`/dev/urandom`) and `stderr` (fatal message).

## Findings

### F-013 (LOW) — `version_from_str` unbounded separator scan

**File:** `src/types.cyr`, `version_from_str`.

**Issue.** The prerelease (`-`) / build (`+`) separator after the patch
segment was located with two `strchr(data + dot2 + 1, …)` calls.
`strchr` walks until a NUL terminator and takes **no `s.len` bound**.
The major/minor dot-finding loops directly above it *are* correctly
bounded by `slen`; the separator scan was the lone exception.

For a `Str` built via `str_new(ptr, len)` as a slice of a larger
buffer — the common case for substrings and parser-internal slices —
the bytes at and after `slen` are not guaranteed to be NUL. `strchr`
therefore reads past the logical end of the string and can locate a
`-`/`+` belonging to adjacent memory, setting `patch_end` beyond
`slen`. The subsequent digit-validation loop and `str_to_int` then read
the over-extended patch segment, yielding either a spurious parse error
or a corrupted patch number for an otherwise-valid version.

**Why it stayed latent.** Inputs via `str_from` (NUL-terminated C
literals) mask the bug, so every unit test passed. The fuzz harness
(`test_v112_fuzz.tcyr`) also missed it: `fuzz_random_bytes` allocates
`len + 1` and explicitly writes a NUL at `buf + len`, so every fuzzed
`Str` is NUL-terminated at exactly its length — the one shape that
hides the over-read.

**Severity rationale: LOW.** Read-only over-read (no write, no
corruption of agnostik state), bounded in practice by the next NUL or
separator in the heap, and not reachable with the NUL-terminated inputs
that dominate real callers. It is a defense-in-depth correctness/buffer
issue, not a remote or escalation vector.

**Fix.** Replaced the two `strchr` calls with a single bounded forward
scan over `[dot2+1, slen)` that stops at the first `-` or `+` (semver
orders prerelease before build, so the first separator is the patch
end). The scan honors `slen` exactly and removes the redundant
double-pass. Uses the flag / `j = slen` loop-exit idiom per CLAUDE.md
(no `break` with an in-loop `var`).

**Regression test.** `tests/tcyr/test_v130_slice_safety.tcyr` constructs
non-NUL-terminated version slices over a larger buffer (all-digit filler
+ a stray `-` past `slen`) and asserts the patch parses as the in-slice
value, not the over-read filler. Two cases: no-in-slice-separator (bound
respected) and in-slice-separator (real prerelease still honored).

## Re-verification

- **F-001..F-012** (2026-04-26 + 2026-05-10 audits) re-checked: all
  remain closed. The hex-decode consolidation this release (single
  `_hex_nibble` routing all UUID/trace/span/JSON hex parsing) preserves
  the F-001 reject-on-invalid behavior — fuzz survival holds.
- **Input validation** — every parser bounds-checks against `s.len`
  before indexing; the F-013 fix closes the last unbounded scan.
- **Buffer safety** — `_proto_*` rewrite to `str_builder_add` delegates
  bounds growth to the stdlib `_sb_grow` + `memcpy`; no new raw writes.
- **Syscall review** — no new syscall surface this release.
- **No command injection / path traversal** — no `sys_system`, no
  file-path handling added.

## Deferred (low-priority cleanups — no action this release)

These surfaced in the review but are either public-API surface (removal
would be a breaking change requiring a major bump) or low-value layout
changes. Tracked in `docs/development/roadmap.md` backlog.

- **Dead/vestigial public helpers** — `seccomp_errno`/`seccomp_trace`,
  the `SeccompArg` cluster, `id_mapping_*`, `network_policy_*`
  (`src/security.cyr`); `stream_usage` (`src/llm.cyr`);
  `AgentInfo_from_json` (`src/agent.cyr`, also cannot round-trip its
  own `_to_json` — emits `agent_type`/`status` name strings but reads
  `agent_type_id`/`status_id` ints). All are tracked in
  `docs/api-surface.snapshot`; removing or fixing them is an API change
  gated on confirming no external consumer depends on them.
- **Setter-less flag getters** — seven `mcap_supports_*` getters
  (`src/llm.cyr`) have no matching setter, so they can only read 0.
  Either incomplete public API (add setters) or decorative.
- **Over-allocated struct** — `secret_metadata_new`
  (`src/secrets.cyr`) allocs 72 B / 9 slots but only 6 have accessors
  (offsets 24/56/64 are zero-init dead space). Shrinking to 56 B is a
  layout change; no raw `store64` writers exist in-repo, but external
  consumers may. Defer until confirmed.

## Conclusion

One LOW buffer-safety finding (F-013), fixed and regression-tested. No
CRITICAL/HIGH/MEDIUM findings. Public API and wire formats unchanged.
Deferred items are cleanup candidates gated on cross-consumer API
confirmation, not security exposures.
